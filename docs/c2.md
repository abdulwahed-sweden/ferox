# Command & Control Layer

Ferox offers safe-by-default C2 helpers for building covert channels, relays, and operator tooling. All modules are scaffolds intended for authorized exercises and research.

## CLI Interface
```bash
ferox c2 list
ferox c2 describe teams_tunnel --verbose
ferox c2 health --config config/c2.example.toml
```

The CLI handler surfaces module metadata, options, and configuration health (including `FEROX_C2_TOKEN` checks and TOML validation).

## Modules
### Teams Tunnel (`c2/teams_tunnel`)
- Uses Microsoft Graph API, meeting descriptions, and chat messages to smuggle commands/results.
- Supports `mock_mode=true` for offline testing.
- Encrypts payloads using derived AES-GCM keys.

### GitHub Gist C2 (`c2/github_c2`)
- Dead-drop channel via private gists, encrypted content, and gist comments.
- Mimics developer API traffic patterns.
- Poll interval and filenames are configurable.

### Relay Manager (`c2/relay_manager`)
- Session fan-out scaffold with Tokio channels.
- Useful for testing scheduling, throttling, or multi-implant flows.

## Configuration
`config/c2.example.toml` demonstrates environment variable-driven tokens, TLS verification, sandboxed file roots, and beacon intervals. Copy and adjust it for each engagement:
```toml
auth_token_env = "FEROX_C2_TOKEN"
allowed_roots = ["./"]
beacon_poll_interval_ms = 1000
cloud_provider = "mock"
```
Set tokens at runtime:
```bash
export FEROX_C2_TOKEN="change_me"
```

## Mocking & Safety
- Every module exposes `mock_mode` or `safe_mode` toggles.
- Ferox Doctor validates dependency readiness (`teams`, `dns`, `python`, TLS support).
- Use dedicated tenant accounts and never point proof-of-concepts at production tenants without contracts.

Extend the C2 layer by registering new modules in the `ModuleRegistry` and documenting them here.
